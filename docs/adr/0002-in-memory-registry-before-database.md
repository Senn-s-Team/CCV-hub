# ADR 0002 - 首版以内存注册表作为运行态真相源

## Status
Accepted

## Context
`ccv-hub` 首版只展示运行中实例，不展示历史实例，不做复杂生命周期管理，不做团队协作，不做云同步。系统设计文档已经明确：本地常驻服务是协调中心，实例注册表是当前运行实例的真相源。产品文档要求实例退出后在可接受时间内从列表移除，同时避免把半状态实例暴露给页面。

在这种边界下，实例状态存储存在两条主要路径：

1. 使用进程内内存注册表作为运行态真相源
2. 首版直接引入数据库，如 SQLite

## Decision
首版采用**进程内内存注册表**作为运行态真相源。数据库持久化后置到增强阶段，仅当收藏、最近打开、轻量审计或更强恢复需求出现时再引入 SQLite。

## Why
- 当前产品范围只关心“现在有哪些实例在运行”，内存模型最贴近这个问题本身。
- 内存注册表可以让 `starting / running / stale / exited / removed` 这些内部状态在单一进程中直接流转，减少跨层同步成本。
- coding agent 在基于内存注册表实现时更容易保持分支收敛，避免首版就被数据库 schema、迁移、持久化一致性拖重。
- 产品当前没有历史查询诉求，数据库在首版中的边际收益很低。

## Consequences
### Positive
- 实现简单，状态模型清晰。
- 服务端可直接承担真相源角色，页面读取链路短。
- 更容易避免半状态实例被写入持久层后再回滚。
- 适合 MVP 快速迭代。

### Tradeoffs
- 服务重启后运行态数据不会自动保留。
- 若未来出现收藏、最近打开、审计等持久化需求，需要补充新的存储层。
- 若实例收敛逻辑变复杂，仍需要谨慎管理内存中的过渡状态与超时清理。

## Rejected alternatives
### SQLite 作为首版主存储
SQLite 适合增强阶段承接历史、偏好和审计类需求，但首版引入它会增加 schema、迁移、初始化、回放与异常恢复复杂度。当前范围下，这部分复杂度不会带来等量收益。

## Implementation notes
- 本地服务持有实例注册表。
- 注册表字段对齐 `docs/api/api.md` 的 `Instance` 模型。
- 只有启动成功且可访问的实例才能进入 `running`。
- 收敛逻辑基于 `lastSeen`、进程退出或探测失败触发。
- 若需要增强服务重启体验，可增加轻量 JSON 快照，作为数据库前的过渡方案。

## Review trigger
出现以下任一条件时，重新评估此决策：

- 产品引入收藏、最近打开或历史实例
- 服务重启恢复成为高频诉求
- 需要更强的审计、追踪或跨会话偏好存储
