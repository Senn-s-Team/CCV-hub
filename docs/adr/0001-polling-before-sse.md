# ADR 0001 - 首版使用轮询作为实例同步主路径

## Status
Accepted

## Context
`ccv-hub` 首版的核心任务是把本机运行中的 `cc-viewer` 实例稳定呈现在总览页中，并支持启动、打开、复制链接与低延迟刷新。现有产品与信息架构文档只要求“自动刷新或低延迟刷新”，同时强调 MVP 优先轻量、稳定、可解释、快速落地。系统边界当前仍限定在单机、本地服务、运行中实例，不包含复杂实时协同能力。

在这种边界下，实例列表刷新存在两条可行路径：

1. 前端定时轮询 `GET /api/instances`
2. 服务端主动通过 SSE 推送实例变化

## Decision
首版采用**前端轮询**作为实例同步的唯一主路径，默认以 2 秒间隔读取 `GET /api/instances`。页面进入后台后自动降频。SSE 进入增强阶段，待首版闭环稳定后再评估接入。

## Why
- 当前文档对刷新体验的要求已经可以被轮询直接满足。
- 轮询链路更短，状态更清晰，页面与服务端更容易对齐错误边界。
- 首版只涉及运行中实例列表，不需要在一开始承担更复杂的服务端推送连接管理。
- coding agent 在实现轮询时更容易保持结构简单，减少额外协议层与连接生命周期分支。

## Consequences
### Positive
- 实现成本低，便于快速交付 MVP。
- 调试路径直观，请求与返回都可被直接观察。
- 更容易把 loading、empty、discovery-error、launch-failed 等状态跑通。
- 前后端职责边界清楚：服务端负责返回 running 列表，前端负责按固定节奏读取。

### Tradeoffs
- 实例变化需要等待下一次轮询周期才能反映到页面。
- 当页面长期打开时，请求会持续发生，需要通过后台降频控制开销。
- 后续若接入更多细粒度实时状态，仍需要新增推送能力。

## Rejected alternatives
### SSE 作为首版主路径
SSE 很适合后续增强阶段的细粒度实时更新，但首版里它会引入额外的连接管理、重连、事件协议、错误恢复与浏览器状态处理。当前产品边界与收益结构更适合先把轮询闭环做稳。

## Implementation notes
- 使用 React Query 管理 `GET /api/instances` 的轮询。
- 默认轮询周期 2 秒。
- 页面后台自动降频到 5~10 秒。
- 页面只消费 `running` 列表，内部状态收敛由服务端处理。

## Review trigger
出现以下任一条件时，重新评估此决策：

- 页面需要展示比当前更细粒度的实例实时状态
- 本地轮询开销明显影响体验
- 公网访问模式要求更稳定的实时链路
